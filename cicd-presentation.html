Module 1

Introduction to Continuous Integration

What is Continuous Integration?

A software development practice where developers frequently integrate code into a shared repository, with each integration verified by automated builds and tests.

Core Principles

  • Frequent Commits: Integrate code multiple times per day
  • Automated Builds: Every commit triggers a build
  • Automated Tests: Run tests on every build
  • Fast Feedback: Know within minutes if something broke

Goal: Detect and fix integration problems early, when they're easiest to resolve

Before CI: Integration Hell

Traditional Approach

  • Developers work in isolation for weeks
  • "Big bang" integration at the end
  • Merge conflicts everywhere
  • Bugs discovered late in cycle
  • Painful, stressful releases

The Problems

  • "It works on my machine!"
  • No one knows if the build is broken
  • Manual testing is slow and error-prone
  • Fear of making changes
  • Long release cycles

Why CI? The Benefits

Quality

  • Bugs found early
  • Consistent builds
  • Automated testing
  • Code review integration

Speed

  • Fast feedback loops
  • Reduced integration time
  • Automated deployments
  • Faster time to market

Confidence

  • Always deployable code
  • Reduced risk
  • Better collaboration
  • Documentation via code

Key Insight: CI transforms integration from a dreaded event into a non-event

CI vs CD vs CD

Continuous Integration (CI)

Automatically build and test code changes

↓
Continuous Delivery (CD)

Automatically prepare releases; manual deployment approval

↓
Continuous Deployment (CD)

Automatically deploy every change to production

Note: Start with CI, then evolve to Continuous Delivery, then optionally to Continuous Deployment

The CI Philosophy

Developer Mindset

  • Commit early, commit often: Small, incremental changes
  • Never break the build: Run tests locally first
  • Fix broken builds immediately: Top priority
  • Write tests: No code without tests
  • Keep builds fast: Under 10 minutes ideally

Mantra: "If it hurts, do it more often" - frequent integration reduces pain

Jenkins Overview

Jenkins is an open-source automation server that enables building, testing, and deploying software.

Key Features

  • Extensible: 1800+ plugins for every need
  • Distributed: Scale with multiple agents
  • Pipeline as Code: Define builds in Jenkinsfile
  • Free & Open Source: Large community support

History: Originally Hudson (2004), forked to Jenkins in 2011. Used by Netflix, Facebook, and thousands of companies.

Jenkins Architecture

Components

  • Controller: Main server, manages jobs
  • Agents: Execute build tasks
  • Executors: Parallel build slots
  • Workspace: Job working directory

Job Types

  • Freestyle: GUI-configured jobs
  • Pipeline: Code-defined workflows
  • Multibranch: Auto-discover branches
  • Folder: Organize jobs

Jenkins Pipeline Basics

Pipelines define your build process as code in a Jenkinsfile.

pipeline {
    agent any

    stages {
        stage('Build') {
            steps {
                echo 'Building the application...'
                sh 'make build'
            }
        }
        stage('Test') {
            steps {
                echo 'Running tests...'
                sh 'make test'
            }
        }
        stage('Deploy') {
            steps {
                echo 'Deploying...'
            }
        }
    }
}

Pipeline Syntax

Declarative Pipeline

  • Structured, opinionated syntax
  • Easier to read and write
  • Built-in validation
  • Recommended for most cases

Scripted Pipeline

  • Full Groovy programming
  • More flexible
  • Steeper learning curve
  • For complex workflows

Best Practice: Start with Declarative syntax; use Scripted only when you hit limitations

Continuous Testing

The Testing Pyramid

E2E Tests
Integration Tests
Unit Tests

Unit Tests (Base)

  • Fast, isolated, many
  • Test single functions/classes
  • Run in milliseconds

Integration/E2E (Top)

  • Slower, fewer
  • Test system interactions
  • More realistic but fragile

Types of Automated Tests

Type Purpose Speed Tools
Unit Test individual components Fast pytest, JUnit, Jest
Integration Test component interactions Medium pytest, TestNG
System Test complete system Slow Selenium, Cypress
Functional Test business requirements Varies Cucumber, Robot
Performance Test speed/load Slow JMeter, Locust

Testing in CI Pipelines

1. Checkout

Get code

2. Build

Compile

3. Unit Test

Fast tests

4. Integration

API tests

5. Report

Results

Best Practices

  • Run fast tests first (fail fast)
  • Parallelize when possible
  • Generate coverage reports
  • Set minimum coverage thresholds

Writing Tests for Defects

When a bug is found, write a test that reproduces it before fixing.

# Test that reproduces bug #123
def test_bug_123_divide_by_zero():
    """
    Bug: Calculator crashes when dividing by zero
    Expected: Should return error message
    """
    calc = Calculator()
    result = calc.divide(10, 0)
    assert result == "Error: Division by zero"

Benefit: The test ensures this bug never returns. Your test suite becomes a regression safety net.

Continuous Quality

Code Inspection vs Code Testing

Code Testing

  • Verifies behavior
  • Runs the code
  • Catches runtime errors
  • Tests what code does

Code Inspection

  • Analyzes code structure
  • Static analysis (no execution)
  • Catches style/pattern issues
  • Tests how code looks

Together: Testing ensures correctness; inspection ensures maintainability. You need both!

Static Analysis Tools

Tool Language Purpose
pylint / flake8 Python Style, errors, complexity
ESLint JavaScript Style, errors, best practices
ShellCheck Bash Shell script analysis
SonarQube Multi-language Comprehensive quality platform
Hadolint Dockerfile Dockerfile best practices

Code Quality Metrics

Common Metrics

  • Cyclomatic Complexity: Number of code paths
  • Code Coverage: % of code tested
  • Duplication: Copy-pasted code
  • Technical Debt: Estimated fix time

Quality Gates

  • Coverage > 80%
  • No critical issues
  • Duplication < 3%
  • Complexity < 10 per function

Tip: Set quality gates that fail the build if standards aren't met

Module 5

Continuous Deployment

Deployment Philosophy

If CI is about integrating code, CD is about getting that code to users safely and reliably.

Key Principles

  • Automation: No manual steps in deployment
  • Repeatability: Same process every time
  • Reversibility: Easy rollback if needed
  • Visibility: Everyone knows deployment status

Goal: Deploying should be boring - a non-event that happens many times per day

Release Strategies

Blue-Green Deployment

  • Two identical environments
  • Switch traffic instantly
  • Easy rollback
  • Requires 2x resources

Canary Deployment

  • Gradual rollout (1%, 10%, 100%)
  • Monitor for issues
  • Rollback if problems
  • Lower risk

Rolling Deployment

  • Update instances one by one
  • No downtime
  • Mixed versions temporarily

Feature Flags

  • Deploy code, toggle features
  • Separate deploy from release
  • A/B testing capable

Database Integration in CI/CD

Database changes are often the hardest part of deployment.

Strategies

  • Migration Scripts: Version-controlled schema changes (Flyway, Alembic)
  • Backward Compatible: New code works with old schema
  • Expand-Contract: Add new → migrate data → remove old
  • Blue-Green DB: Separate databases for each environment

Rule: Never make breaking schema changes. Always add, then remove in a later release.

Fast Feedback is Critical

  • Build notifications (Slack, email)
  • Test result dashboards
  • Deployment status visibility
  • Production monitoring alerts

CI/CD Metrics to Track

Metric What It Measures Target
Build Duration Time from commit to result < 10 minutes
Build Success Rate % of builds that pass > 95%
Deployment Frequency How often you deploy Daily or more
Lead Time Commit to production < 1 day
MTTR Mean time to recover < 1 hour
Change Failure Rate % deployments causing issues < 15%

DORA Metrics: These four metrics (deployment frequency, lead time, MTTR, change failure rate) are industry standards

CI/CD Best Practices

Build

  • Single source of truth
  • Reproducible builds
  • Version everything
  • Keep builds fast

Test

  • Test early, test often
  • Maintain test quality
  • Don't skip failing tests
  • Parallelize tests

Deploy

  • Automate everything
  • Deploy to production-like envs
  • Feature flags over branches
  • Always be able to rollback

Rolling Out CI in Your Organization

Getting Buy-in

  • Start Small: Pilot with one team or project
  • Show Value: Track and share metrics improvements
  • Train Teams: Invest in education
  • Iterate: Continuously improve the process

Culture Shift: CI/CD is as much about culture as tools. Blameless postmortems, shared ownership, and continuous learning are essential.
